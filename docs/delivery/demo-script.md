# 录屏脚本

## 1. 需求理解

说明本题重点不是 UI，而是 AI 工具使用、MVP 拆解、核心流程可运行和边界清楚。

## 2. MVP 定义

第一版做内部员工助手：

- 普通聊天。
- 多轮上下文。
- 自动工具调用。
- mock 模式。
- 更丰富的 mock 内部员工场景：政策、福利、入职、采购、会议室、安全合规、计算。
- 本地可启动。
- 基础安全与并发保护作为加分项，但不把 MVP 复杂化。

不做完整登录、数据库、RAG、流式输出和生产级权限；但预留 SSO/JWT、Redis 限流、Redis/Postgres session 和 LLM 网关治理的接口。

## 3. AI 协作过程

展示如何让 AI 帮助：

- 拆解需求。
- 设计目录。
- 生成实现提示词。
- 审查并收敛范围。
- 修复构建或测试问题。

## 6. Mock 模式

说明没有 `.env` 和 API Key 时仍默认 mock 模式运行。

## 6.1 安全与并发拔高点

建议用 1-2 分钟讲清楚“哪些已做基础版、哪些只是生产化接口”：

- 后端运行时校验 `message` 和 `sessionId`，并限制请求体大小。
- 可选 `API_BEARER_TOKEN` 只作为演示认证口子，真实内部系统应替换为 SSO/JWT/OIDC。
- 进程内 IP、用户、session 三层限流，生产替换为 Redis 或 API 网关。
- 真实 LLM 模式有全局和单 session 并发闸门，避免大量请求打爆 DeepSeek、中转站或内部网关。
- 内存 session 有 TTL、最大 session 数和每 session 最大消息数，生产替换为 Redis/Postgres。
- API 统一错误结构和基础安全响应头，前端只展示安全错误文案。

## 7. 限制和后续优化

说明当前限制：

- 内存会话重启丢失。
- 工具路由是演示级规则。
- 真实模型模式未做完整 function calling。
- 认证、限流、session 和 LLM 并发控制是本地基础版，不是完整生产级。

后续优化：

- 数据库持久化。
- Redis/API 网关限流。
- SSO/JWT/OIDC。
- LLM provider 成本、配额和熔断治理。
- 真实 function calling。
- 权限和审计。
- 流式输出。
